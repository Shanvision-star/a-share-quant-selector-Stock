# 前端加载性能与参数保存可靠性重构设计（深度重构方案）

## 1. 问题定义

### 1.1 业务痛点
- 前端页面（尤其首页、策略结果页、个股详情页）加载慢，首屏响应不稳定。
- 参数设置页在用户未点击“保存”时切换页面会丢失编辑内容，缺少“未保存离开提醒”。
- 需要在不改变“必须手动保存”的业务规则下，提升可用性与性能。

### 1.2 本次目标
- 建立清晰的前后端分层边界，降低模块耦合。
- 统一页面查询状态与请求编排，降低重复请求与无效渲染。
- 参数设置支持“草稿保留 + 离开提醒 + 显式保存”。
- 给出分阶段优化路径、修改文件清单、风险与缓解策略。

### 1.3 非目标
- 本轮不改动策略算法本身（B1/B2/碗口逻辑）。
- 不引入新的数据库类型；继续使用现有 CSV/JSON/SQLite 体系。

---

## 2. 现状架构问题梳理

### 2.1 模块耦合问题
1. `SettingsView.vue` 直接持有参数编辑态并直接调用接口，缺少跨路由可复用状态层，导致离开页面即丢失未保存编辑。
2. `HomeView.vue`、`StrategyResultsView.vue` 各自维护分页/筛选/排序与加载流程，存在重复逻辑，页面间无法复用查询状态。
3. 后端 `stock.py` 路由承担大量计算与缓存编排逻辑，Router 与领域逻辑耦合高，维护和回归成本高。
4. 配置保存与策略注册器刷新之间契约不清晰：配置文件更新后，运行时策略参数刷新机制没有显式一致性保证。

### 2.2 接口设计问题
1. `/api/config` 仅“读全部/写参数”，缺少修订版本（revision）与冲突保护。
2. 错误语义较粗（例如仅 400），前端难以区分“并发覆盖冲突”“参数校验失败”“内部异常”。
3. 页面请求发起策略偏“视图驱动”，缺少统一去重、取消和重试策略。

### 2.3 性能瓶颈
1. 页面挂载时并行或串行触发多个接口，缺少统一调度，造成抖动。
2. 表格和图表页面请求没有统一取消机制，切页时可能仍在请求和渲染。
3. 个股详情页聚合多接口与大体量 K 线渲染，首屏压力高。
4. 后端股票列表接口虽然已有快照与缓存，但逻辑集中在路由层，扩展优化难。

---

## 3. 目标指标（按现状评估后给出的建议）

- Phase 1 目标：首页/策略结果页首屏 **P95 <= 2.5s**。
- Phase 2 目标：首页/策略结果页首屏 **P95 <= 2.0s**。
- 参数配置可靠性目标：
  - 未保存离开提醒命中率 100%；
  - 同会话内切页返回后草稿不丢失；
  - 保存成功后刷新页面应可读到最新参数值。

---

## 4. 目标架构设计

## 4.1 分层与职责
1. **View 层（页面/组件）**
   - 仅负责展示与交互事件，不直接拼接复杂请求参数和缓存策略。
2. **PageModel/Store 层**
   - 统一管理页面查询状态（分页、筛选、排序、日期）；
   - 管理设置页草稿、dirty 状态、离开拦截。
3. **API Client 层**
   - 统一请求去重、取消（AbortController）、错误映射、接口契约封装。
4. **Backend Service 层**
   - 将 `router` 中的重逻辑下沉到 `services`，Router 仅做参数校验与编排调用。

## 4.2 参数保存语义（保持“手动保存”）
- 用户调整参数后只更新“草稿态”，不立即写后端。
- 路由离开、关闭标签页、切换关键页面时触发 dirty 检查。
- 用户点击“保存”时提交 `expected_revision`：
  - 成功：落盘、刷新 revision、清除草稿；
  - 冲突：返回 409，提示“配置已被其他操作更新，请刷新后重试”。

## 4.3 数据流（设置页）
1. `GET /api/config` -> 初始化 `serverSnapshot + draft + revision`
2. 用户改值 -> 更新 `draft`，标记 `dirty = true`
3. 路由离开 -> Guard 检查 `dirty`，弹窗确认
4. 点击保存 -> `POST /api/config`（带 `expected_revision`）
5. 保存成功 -> 同步 `serverSnapshot = draft`，`dirty = false`

## 5. 分阶段优化方案

## Phase 1：可用性止损与请求治理

### 目标
- 快速解决“参数切页丢失”与明显无效请求问题。

### 方案
1. 新增设置页草稿 Store（会话级持久化）与 dirty 标记。
2. 增加路由离开守卫（含浏览器 `beforeunload`）进行未保存提醒。
3. 为首页、策略结果页、个股页接入请求取消与去重（切页/筛选快速切换时）。
4. 统一前端错误消息映射（网络、超时、冲突、服务错误）。

### 验收
- 参数调整后切页返回仍在草稿态；
- 未保存离开必弹提醒；
- 快速切页不再积压旧请求回写 UI。

## Phase 2：核心架构重整

### 目标
- 降低页面与接口耦合，形成可持续演进的查询与配置模型。

### 方案
1. 抽取通用 QueryModel（分页/筛选/排序/日期）供首页与策略页复用。
2. `/api/config` 增加 revision 与冲突返回码（409）。
3. 后端将配置读写、列表聚合等逻辑从 Router 下沉到 Service。
4. 统一接口 DTO（请求/响应字段命名与错误结构）。

### 验收
- 关键页面查询状态在路由切换后按设计保留；
- 配置并发写冲突可被识别且可恢复；
- Router 文件显著瘦身，职责清晰。

## Phase 3：深度性能优化

### 目标
- 达到更稳定的首屏与交互响应。

### 方案
1. 股票列表接口拆分“轻量列表字段”与“详情扩展字段”，避免不必要数据传输与计算。
2. 策略结果查询优化：索引复核、分页查询字段裁剪、热点筛选组合优化。
3. 图表页优化：可见区优先渲染、延后次要面板数据请求。
4. 建立性能基线采集（前端关键接口耗时 + 页面首屏指标）。

### 验收
- 首页/策略页首屏达到目标；
- 图表交互掉帧和卡顿明显下降；
- 查询接口 P95 可观测并持续收敛。

## 7. 风险提示与缓解策略

1. **风险：深度重构引发回归面扩大**
   - 缓解：按 Phase 逐步落地，保持每阶段可独立回滚。

2. **风险：参数 revision 引入后出现兼容性问题**
   - 缓解：提供向后兼容分支（未传 expected_revision 时按旧逻辑并记录告警）。

3. **风险：策略注册器刷新不完整导致“保存后不生效”**
   - 缓解：定义保存后强一致刷新路径（重载 params + 重建策略实例或受控热更新）。

4. **风险：请求取消/去重处理不当导致页面数据不更新**
   - 缓解：统一请求状态机，明确“最新请求胜出”规则，并添加集成测试。

5. **风险：列表接口性能优化影响字段完整性**
   - 缓解：明确列表字段契约与详情字段契约，前端按契约分层消费。

---

## 8. 评审结论

- 该方案保持“手动保存”业务规则不变，优先修复参数可靠性，再完成架构解耦与性能深度优化。
- 采用深度重构路线，但通过分阶段切分交付面，降低一次性改动风险。
